This document is a short tutorial that aims to introduce you to a typical UNIX-like environment, as the one you can find on CSIF computers.

Make sure to read it if you don’t feel comfortable using the terminal, connecting remotely to other UNIX computers using SSH, etc.

Preamble

In the early 1970s, pioneering engineers at AT&T Bell Labs began developing a small operating system that they called Unix. Initially written entirely in assembly language, it was rewritten a couple of years later in a new high-level language, especially developed for the occasion by the same group of engineers, called C. This programming language switch enabled Unix to be easily portable to new computer hardware, which reinforced its already growing popularity.

Because AT&T was not yet authorized to sell Unix directly as a product, Unix started to be licensed to outside parties from the late 1970s. Throughout the 1980s, many academic and commercial variants of Unix were harshly competing: amongst the strongest competitors (with their own Unix variants) were UC Berkeley (BSD), Microsoft (Xenix), IBM (AIX) and Sun Microsystems (Solaris).

In parallel, Richard Stallman from the MIT started the GNU project, which aimed to create a completely free and Unix-compatible operating system. By the early 1990s, the project already counted numerous fundamental software components, many of which we will use in this class, including a C compiler (gcc), a debugger (gdb), a C standard library (libc) and dozens of other utilities. The only missing part, to make a complete operating system, was the kernel.

Incidentally, in the summer of 1991, a Finnish student named Linus Torvald began developing his own kernel, which he later called Linux, and made it available for anyone to use and modify. It sparked a huge interest from the community and by the mid-1990s, the Linux kernel combined with the GNU components and other tools could be distributed as a complete and functional operating system.

Today, existing Unix-like (aka *nix) operating systems embrace:

  • A handful, mostly closed-source, commercial survivors directly derived from the original AT&T Unix, such as AIX, Solaris or HP-UX.
  • Some open-source and free distributions based on UC Berkeley’s BSD branch, such as FreeBSD, NetBSD or OpenBSD.
  • Mac OS X, which also derive from the BSD branch and for which some parts of the operating system are open-source.
  • All open-source and free GNU/Linux distributions.

Almost all of these computers are accessible to students registered for computer science classes at UC Davis. For this class, the CSIF computers are considered the reference environment in which your assignments will be graded.

In case of issue of if you have any questions regarding CSIF, you can go to the support office located in Academic Surge, room 1114, or write an email at support@cs.ucdavis.edu.

Logging in

If you are a CS major or are taking a CS class, you should automatically get a CSIF account which enables you to log into any of the CSIF computers. To access your account, you need to know your UC Davis Kerberos username and password (that is the same username/password you are using to access your email account).

There are two options to log into the CSIF computers: locally in the basement labs, or remotely from your own computer and through the network.

Local

Physically go to one of the basement labs in Kemper Hall. The labs are located in rooms 67, 71 and 75. Choose a computer that is available and appears to be working. Enter your username and password at the login screen. If everything goes well, the desktop environment should be loaded and you can start running applications.

In order to start a terminal, click on the menu (top icon in the left sidebar), search for terminal and open the terminal application. The terminal is the window in charge of the graphical display. Internally, it runs another (non-graphical) application called the shell which processes input commands.

The shell shows a command line prompt indicating that it’s ready to receive commands and execute them. Such prompt usually shows your username, computer name, the path to the current working directory and ends with the character $. On the same line as the prompt you can type in a shell command and press Enter to execute it.

Remote

Because of many attacks over the summer of 2020, the College of Engineering has put the CSIF behind a VPN.

Follow the instructions in the following page in order to set up a VPN connection on your computer: http://csifdocs.cs.ucdavis.edu/news/vpnrequiredtoconnecttocsifcomputers

The CSIF computers are remotely accessible through the network via a protocol called SSH (Secure SHell). The computers you are authorized to access are named pc01 to pc43. You can view the real-time status of all these computers on this page. Pick a computer to which you would like to get connected (take a computer noted as “UP” in the status report since computers noted as “DOWN” are not reachable).

Each of these computers are publicly accessible from outside campus through their full public address. Such address is obtained by appending .cs.ucdavis.edu to the computer’s name. For this tutorial, we choose computer named pc42 which full public address is then pc42.cs.ucdavis.edu.

If your computer runs a GNU/Linux distribution or if you have a Mac computer running Mac OS X, then the SSH software is probably already available. If your computer is running Windows, jump down to the related paragraph in order to install the necessary software.

GNU/Linux or Mac

In order to connect to computer pc42, open a terminal window and at the shell prompt, type in the following command. Note that <user> has to be replaced by your Kerberos username.

$ ssh <user>@pc42.cs.ucdavis.edu

If you have never connected to the computer before, SSH will ask if you are sure of the computer’s authenticity:

The authenticity of host 'pc42.cs.ucdavis.edu (128.120.211.123)' can't be established.
ECDSA key fingerprint is SHA256:bzT3S4HdKczDmnkim5BdxfTig9LRgxd8ipT7DulO8xg.
Are you sure you want to continue connecting (yes/no)?

In such case, type in yes in order to continue connecting. Once the authenticity of the computer is established, SSH will prompt you for your password on the remote computer. Enter your Kerberos password.

Once you are connected to the remote computer, you should have access to a shell prompt and be able to enter commands.

Windows

Since Windows does not come with SSH installed, you need to download and install an SSH client yourself. PuTTY is an open-source SSH client that can be downloaded from this page. Install one of the MSI installers, or just grab one of the putty.exe binary executables.

When running PuTTY, a dialog window will appear. Enter the full public address of the computer to which you would like to connect (e.g. pc42.cs.ucdavis.edu) in the Host Name box. Make sure that the Port is set to 22, and that the protocol is set to SSH. If you wish, you can save this configuration for later use by entering a descriptive name (e.g. “CSIF - pc42”) in the Saved Sessions box and clicking Save.

Now click on Open at the bottom of the dialog, in order to start the connection. If you have never connected to the computer before, PuTTY will display a PuTTY Security Alert dialog box that asks if you trust this host. Click Yes to accept.

After entering your Kerberos username and password, you should be connected to the remote computer, have access to a shell prompt and be able to enter commands.

Navigating the file system

Absolute pathname

Each directory has an absolute pathname that uniquely specifies the location of that directory within the computer’s file system. As you move within the directory structure, you can use the command pwd in order to display the absolute pathname of the current location. The pwd command actually stands for “print working directory”.

Type pwd in the shell now. It should display /home/<user>, with <user> being your Kerberos username (e.g., /home/jporquet). This directory is your home directory. This is the place where your data are located in a typical *nix file system. Each user has their own home directory that is completely under their control.

Change directory

The command cd, which stands for “change directory”, allows you to move to any other accessible directory in the file system. The syntax for cd is: cd [path_name]. Now, change to the /bin directory by typing cd /bin. After executing this directory change, use pwd to confirm that you now are in the /bin directory.

List files

The command ls, which stand for “list”, allows you to see the contents of a directory. The syntax for ls is: ls [-al] [pathname], where the square brackets (“[]”) indicate optional values. For now, simply type ls.

You will see a list of all the file names that are included in the /bin directory. This directory holds many of the basic *nix utilities, some of which you will get to discover later in this tutorial or during this quarter. Note that command ls is listed, but command cd, for example, is not: this is because ls is an independent application, while cd is a built-in command provided directly by the shell.

Putting it together

Before you dive into the options of ls, go back to your home directory. You could type cd /home/<user> (e.g., cd /home/jporquet), but there is a better shortcut. Simply type cd, it will bring you back to your home directory automatically.

After the shell has executed the change directory command, confirm that you are back inside your home directory with pwd. Now type ls. If you have not created any files or directories in your account so far, then nothing will be displayed.

Hidden files

Many shell commands have options that may follow the command after a space and a dash. Options are used to modify the command’s behavior. For ls, the options change what you see – giving you specialized types of information which you would not see using the basic command.

Now type ls -a. The -a option stands for “all”. Here is a possible output:

$ ls -a
.               .bashrc
..              .cache
.bash_history   .ssh

The reason you didn’t see these files before specifying the -a option is that their name start with a period which turns them into hidden files. Such files are hidden because they are often configuration or data files belonging to applications and were not directly created by the user.

In the listing, there are two unusual entries: . and ... These entries have a special meaning in terms of pathnames. . stands for the current directory while .. stands for the parent directory.

Type cd . followed by pwd. As you can notice, you have not moved within the directory structure. Now, type cd .. followed by pwd. You should see /home, which means you changed to the parent directory.

Long-format listing

Now return to your home directory using cd, and type ls -al. The -l option stands for “long”, and will make ls display a line of information for each entry. Here is an example of possible output:

$ ls -al
total 996
drwxr-xr-x     3 jporquet users   4096 2017-11-13 09:13 .
drwxrwxrwx 12607 root     root  995328 2017-11-09 22:30 ..
-rw-------     1 jporquet users     71 2017-01-22 17:36 .bash_history
-rw-------     1 jporquet users     14 2017-04-05 14:04 .bashrc
drwx------     2 jporquet users   4096 2017-10-23 11:38 .cache
-rw-r--r--     1 jporquet users    416 2016-12-12 13:52 .login
-rw-r--r--     1 jporquet users      6 2016-12-12 13:52 .logout
drwx------     2 jporquet users   4096 2017-11-08 16:25 .ssh

Here is a brief introduction to the information displayed in the long format (from left to right):

  • File type and file permissions field
    • drwxr-xr-x
      • The d at the very left indicates the entry is a directory.
      • The next group of three letters rwx indicates that the owner has read, write and execute permissions for this entry. For a directory, the execute permission means that I can cd into that directory. Read permission means that I can use ls to see the directory’s contents. Write permission means that I can create additional entries (subdirectories or files) into this directory.
      • The next two groups of three show that users of the group named users as well as other users not beloning to that group, are only allowed read and execute permissions.
    • -rw-r--r--
      • The dash at the very left indicates the entry is a regular file.
      • Here the read and write permissions mean the ability for the owner to look at the file (e.g. using commands like less as shown later) or to edit the file (e.g. using a text editor such as vim).
      • Other users can only look the file’s content but not modify it.
  • Links field: number of links to this entry from other directories
  • Owner field: name of owner
  • Group field: name of group
  • Size field: file size in bytes
  • Date field: date stamp at which the file was last modified
  • Name field: entry’s name

Root directory

So far, we haven’t specified any pathname after ls. This caused the contents of the current working directory (as reported by pwd) to be listed. But if you indicate a pathname after the command, you will see its contents instead. Type ls / in order to see all the files and directories that are contained in the file system’s root directory.

The root directory (represented by a sole slash, /) is the top directory in the *nix file system structure, and thus contains all the files and directories of the system. You can notice that it contains the home subdirectory, which holds the home directory of all the users.

Directories

Create

In order to create a new directory, use the mkdir command, which stands for “make directory”. Its syntax is mkdir {dirname}+, where the curly brackets followed by the plus sign indicate that there must be at least one directory name specified.

Now type mkdir first, followed by ls -l. You should see a new directory named first in the file listing. Now type cd first to move into directory first. Use pwd to confirm where you are.

Now type mkdir second third, followed by ls -a. The output should display:

.  ..  second  third

Now type ls -l, and the output should display:

total 8
drwxr-xr-x 2 jporquet users 4096 2017-11-17 02:48 second
drwxr-xr-x 2 jporquet users 4096 2017-11-17 02:48 third

Delete

The command rmdir, which stands for “remove directory”, allows you to delete an empty directory. Its syntax is rmdir {dirname}+.

Now type rmdir second to remove the directory named second. Confirm that the directory was removed using ls. You should now see only third being listed.

Files

Use pwd to make sure that you are still in /home/<user>/first.

Copy

There a two possible syntaxes for the cp command, which stands for “copy”:

  1. cp {source_filename} {destination_filename}, to make a copy of the source file into the destination file
  2. cp {source_filename}+ {destination_directory}, to copy a list of files into another directory

Now you will copy an existing file into the current directory. Type cp /usr/share/common-licenses/GPL-3 myfile.

Use ls to confirm that the file was properly copied under the name myfile. The only other entry should be directory third.

Now copy this file into directory third by typing cp myfile third. You could check that the file copy was successful by cd’ing into the third directory and then using ls but there is a simpler way. Just type ls third in order to display the contents of the third directory. You should see only myfile listed.

Move

The command mv, which stands for “move”, has three possible syntaxes:

  1. mv {old_filename} {new_filename}, to change the name of a file
  2. mv {filename}+ {directory_name}, to move files into another directory
  3. mv {old_directory_name} {new_directory_name}, to change the name of a directory

You will try all three methods in order.

Now type mv myfile wombat to rename file myfile into wombat. Use ls to check your work: you should see only directory third and file wombat.

Now type mv third/myfile . in order to move file myfile to the current directory. Again, use ls to verify that the current directory now contains directory third and files myfile and wombat.

Now type mv third fourth to rename directory third into fourth. Use ls to see directory fourth and files myfile and wombat.

Delete

The syntax of the command rm, which stands for “remove”, is: rm {filename}+.

Type rm myfile and use ls to check that file myfile was removed, leaving only directory fourth and file wombat.

Command-line interface

By default, your CSIF account is set up to use the very popular shell Bash. This shell comes with many features that can greatly help improve the command line workflow.

Edition

Let’s imagine you wish to copy file wombat into directory fourth, but you mistakenly type cp woombat forth/. Fortunately, before you hit the return key to execute the command, you notice your errors.

You can go back to make the appropriate changes by using the left and right arrow keys to move within the command line. You can then correct both errors, adding the missing u in forth and deleting the extra o in woombat.

When you are done correcting the command line, you can press enter to execute the command.

Note that the cursor does not need to be at the end of the command line to press enter.

History

Each time a command is entered, it is added to the shell’s history, which contains a list of all the previously entered commands. You can scroll backward through previous commands by entering the key combination <Ctrl-p> or by using the up arrow key. Inversely, you can scroll forward with key combination <Ctrl-n> or by using the down arrow key.

When you go back to a previous command, you can either just execute it again by pressing enter or edit it using the left and right arrow keys.

If you know approximately the command you would like to go back to, you can also press key combination <Ctrl-r>, which invokes a reverse interactive search. Start typing part of the command and you will get the most recent command of the history that matches your input.

Completion

In order to save typing and time, Bash gives the possibility to complete the various parts of a command.

Let’s imagine you wish to move file wombat into directory fourth. The command would be mv wombat fourth. However, rather than typing the whole command, you can start typing mv w and then hit the <Tab> key. Since file wombat is the only file in the current directory that starts with the letter w, Bash can complete its name for you. You can then continue entering the command by typing f and hit <Tab> again. Since fourth is the only item starting with an f, Bash can also complete its name, eventually giving the complete command mv wombat fourth/.

As you can notice, completion saved 50% keystrokes in this case (16 keystrokes for the full command vs. 9 keystrokes when using completion)!

If several files start with the same letters, Bash might not be able to complete a entire filename based on your partial input. In this case, Bash will try to complete as much as possible and will then wait for you to enter one or more letters in order to choose between the different possibilities. When Bash is waiting for extra characters in order to solve the ambiguity, press <Tab><Tab> in order to list the completion possibilities.

Display

There are several utilities that can be used to display file contents.

In order to display one or several files directly in the terminal, you can use the cat command, which stands for “concatenate”. Its syntax is: cat {filename}+.

Now type cat wombat and observe how the contents of the file just flew by on the screen.

For short files, cat might be fine, but often you will wish to see the file progressively. For that, you can use the command less, a pager application allowing you to scroll through a file. Its syntax is less {filename}+.

Now type less wombat. You will see that only a screen-worth of the file is displayed, starting from the beginning of the file. Less shares a lot of shortcuts with vim. To scroll forward one line, you can either use the down arrow or press j. To scroll up, use the up arrow or press k. To quit, press q.

Sometimes it can happens that you want to either see the first few lines or the last few lines of a file. For that, you can respectively use the command head or the command tail. Their syntax is: head/tail [-n] {filename}+, where option -n is the number of lines to be displayed (default is 10).

Now type head -4 wombat to see the first four lines of file wombat. Type tail -1 wombat to see the last line of file wombat.

Process killing

In most shells, including Bash, it is possible to send special commands to the shell itself using key combination based on the <Ctrl> key. We have seen that key combinations <Ctrl-p> and <Ctrl-n> can be used to scroll through the command history.

Another very important key combination is <Ctrl-c>. It can be used to instantly terminate the process that is currently being executed by the shell.

For example, the ping utility checks the availability of a remote computer by sending periodical ping requests that the remote answers (or not if it’s down). By default, this utility never stops by itself and continues to send ping requests until it’s interrupted by <Ctrl-c>. The syntax of the ping utility is ping {computer_address}.

Now type ping google.com. Your screen will start filling with lines indicating the quality of the network connection between your current host and google.com. Terminate the process by pressing <Ctrl-c>. Ping should display a few lines of statistics after it has received your interrupt signal, and you should get back to the shell prompt again.

You can use <Ctrl-C> in most situation when a command or a process seem to be stuck or perform in an unsatisfactory manner.

Text editors

Although terminals can only offer text-based user interfaces, they still permit to use some of the most powerful text editors for writing programs. Three popular text-based editors available on CSIF computers are Nano, Emacs and Vim.

  • Nano is by far the most simple and less powerful text editor; it is only recommended for small and quick file edits.

  • Emacs is the official text editor of the GNU project and is characterized by its extensibility. It offers a graphical user interface when being used locally, but can revert to a text-based interface when being used in the terminal (over SSH for example).

  • Vim (and by extension, Vi, the original text editor of which Vim is a improved clone) has the advantage of being available on almost every *nix computer. It is also known as the other contender in the infamous Emacs/Vim war. Although Vim only runs in the terminal, Gvim offers a graphical user interface when used locally.

It is strongly recommended to learn how to use a powerful text editor, such as Vim or Emacs, as it would surely increase your productivity. (As well as making you feel like a real hacker!)

For example, in order to get started with vim, type vimtutor in a terminal and follow the tutorial entirely.

Manual pages

In *nix systems, commands are traditionally thoroughly documented using manual pages that are accessible through the man command. For example, in order to know more about ls, you can type in man ls.

Here is a small portion of what you will get:

NAME
       ls - list directory contents

SYNOPSIS
       ls [OPTION]... [FILE]...

DESCRIPTION
       List  information  about  the  FILEs  (the current directory by default).
       Sort entries alphabetically if none of -cftuvSUX nor --sort is specified.

       Mandatory arguments to long options are mandatory for short options too.

       -a, --all
              do not ignore entries starting with .

       -A, --almost-all
              do not list implied . and ..

       --author
              with -l, print the author of each file
...

As you can see, a man page always starts with a very short description of the command and a summary of its syntax. It is typically followed by other sections including the detailed list of all possible options.

As man typically uses the pager less to display its output, the same shortcuts are available to scroll through the documentation (down and up arrows, or k and j) or to quit (q).

Since a command’s manpage is typically meant to be a comprehensive documentation for that command, it can often be a bit overwhelming. There is another tool called tldr which condenses the usage of a command in a few, well-chosen examples. Type tldr ls to see the examples associated to using ls.

Logging out

You can now log out from the CSIF computer, as explained below, and resume a normal activity!

Local

Close the terminal by typing exit at the shell prompt. Close any other running applications. Click on the gear icon located at the far right of the top panel and select “Log Out…”. The computer should log you out and return to displaying the initial login screen.

Remote

In the terminal window, at the shell prompt, type in exit. If you are running PuTTY on windows, this should close PuTTY entirely. If you are running a terminal on GNU/Linux or Mac OS X, it should simply close the connection with the remote CSIF computer but leave the terminal window open. You can either close the window or type in exit again in the terminal to close it.

Appendix

Other typical *nix commands

There are many (powerful) command line utilities that you might find useful at some point during the quarter. Read their manual page in order to understand how to use them.

Here is a small list:

  • zip: compress files or directories into .zip archives
  • grep: look for a text pattern within files
  • find: search for files in a directory hierarchy
  • diff: compare two files and print the differences
  • sort: sort the contents of a file
  • ps: print the list of running processes
  • kill: kill a certain process
  • chmod: change the permissions of a file
  • chown: change the owner of a file
  • Etc.
